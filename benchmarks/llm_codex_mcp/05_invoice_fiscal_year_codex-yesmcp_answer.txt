`InvoiceNumberService` is the service that generates server-side invoice numbers. Its main method is `generateInvoiceNumber(...)`, and it is called from order creation and refund/void flows in [`InvoiceNumberService.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/InvoiceNumberService.php#L167), [`Order.php`](/Users/naman/Documents/QBApps/qbapi/app/Order.php#L10895), and [`Order.php`](/Users/naman/Documents/QBApps/qbapi/app/Order.php#L16542). Fiscal-year rollover itself is orchestrated by `FiscalYearManager`, mainly through `executeDailyFiscalYearCheck`, `processChainFiscalYearChange`, `applyFiscalYearPrefixes`, and `resetServerGeneratedInvoiceSequences` in [`FiscalYearManager.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/FiscalYearManager.php#L50), [`FiscalYearManager.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/FiscalYearManager.php#L140), [`FiscalYearManager.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/FiscalYearManager.php#L417), and [`FiscalYearManager.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/FiscalYearManager.php#L584).

Fiscal-year transition changes numbering by switching the sequence bucket, not by continuing one global counter. `generateInvoiceNumber` looks up the sequence row by `(storeID, isNoCharge, isRefund, resetDate)`. Before the reset date is active, it uses sentinel `resetDate = '1970-01-01'`; once `storeTransactionResetDate <= today`, it uses the fiscal-year reset date instead. That schema is enforced by the unique key in [`2025_11_06_000001_create_invoice_sequence_table.php`](/Users/naman/Documents/QBApps/qbapi/database/migrations/2025_11_06_000001_create_invoice_sequence_table.php#L33). On rollover, `FiscalYearManager` writes the new `storeTransactionResetDate` to stores, applies the fiscal-year prefixes, and seeds or resets new sequence rows for tomorrow with `lastInvoiceNumber = 0`. So when the new fiscal year starts, the first invoice becomes sequence `1` under the new prefix; old fiscal-year rows remain intact rather than being overwritten.

The sequence counter behavior at new FY is therefore: old counter stays in the old `resetDate` row, new FY gets a fresh row with `lastInvoiceNumber = 0`, and the first successful generation increments it to `1`. For device-side fiscal counters, `resetDeviceInvoiceNumbers` clears and reseeds `globalResetInvoice` with `sequenceStart = tomorrow` for six transaction types, and devices are forced to sync at midnight via `forceSyncTime` in [`FiscalYearManager.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/FiscalYearManager.php#L257) and [`FiscalYearManager.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/FiscalYearManager.php#L313).

Compliance checks are fairly strict. The system enforces valid input/date format and required prefix fields in [`InvoiceNumberService.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/InvoiceNumberService.php#L72) and [`InvoiceNumberService.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/InvoiceNumberService.php#L107). Fiscal-year prefix config must use `YYYY-MM-DD`, store-prefix chains must save prefixes per store, and duplicate store prefixes for the same fiscal year are rejected in [`FiscalYearManager.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/FiscalYearManager.php#L671). On the device/compliance side, posting is blocked if fiscal settings are not synced, order IDs must match the active fiscal prefix, and on fiscal-year start day invoice numbers above `500` are rejected in [`Order.php`](/Users/naman/Documents/QBApps/qbapi/app/Order.php#L8107). One nuance: `isNoInvoiceSale` changes prefix selection for refunds, but it is not part of the sequence key, so refund-without-invoice shares the refund counter.

Edge cases:
- Midnight transition: rollover is pre-seeded for `tomorrow`, but `generateInvoiceNumber` switches only when `today/posDate >= resetDate`, so invoices before midnight still use the old bucket and after midnight use the new one.
- Concurrent requests: existing rows use `lockForUpdate`; missing rows use atomic `INSERT ... ON DUPLICATE KEY UPDATE`; deadlocks/timeouts are retried with backoff in [`DatabaseManager.php`](/Users/naman/Documents/QBApps/qbapi/app/Services/DatabaseManager.php#L25).
- Missing chain sequence table: rollover code auto-creates it from the chain1 reference schema before seeding.
- No active devices: server-side fiscal rollover still happens; device resets are just skipped.
- Boundary sensitivity: server generation trusts `posDate` when provided, and fiscal rollover logic uses `Carbon::tomorrow()` without chain-timezone handling, so timezone/date mismatches around midnight are a real risk in this implementation.
