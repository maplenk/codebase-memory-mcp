Assuming this is a new tender code like `PAYMENT_BNPL`, not just a UI label. In this codebase, the write path is mostly data-driven, so the core work is registration plus a sweep of hardcoded reporting/export mappings.

**Registration**
- Add the new payment type in [`cloud_schema/globals.sql`](/Users/naman/Documents/QBApps/qbapi/cloud_schema/globals.sql#L1931) as a `globalTypeValues` row with `typeID = 7`.
- Map it to an account type in [`cloud_schema/globals.sql`](/Users/naman/Documents/QBApps/qbapi/cloud_schema/globals.sql#L1821) via `globalTypePaymentAccountIDMapping` so the POS can resolve the accounting side.
- Enable it per store in [`cloud_schema/store.sql`](/Users/naman/Documents/QBApps/qbapi/cloud_schema/store.sql#L343) using `storeN_paymentSettings`.
- The APIs that expose/add/update store payment settings are [`app/Http/routes.php`](/Users/naman/Documents/QBApps/qbapi/app/Http/routes.php#L2954), [`app/Http/Controllers/adminController.php`](/Users/naman/Documents/QBApps/qbapi/app/Http/Controllers/adminController.php#L2069), and [`app/Admin.php`](/Users/naman/Documents/QBApps/qbapi/app/Admin.php#L7213).
- The runtime payment list shown to the POS is built from `paymentSettings + globalTypeValues + payment/account mapping` in [`app/Store.php`](/Users/naman/Documents/QBApps/qbapi/app/Store.php#L640).
- If channels use allowed payment-mode lists, update channel mappings too in [`app/QBChannel.php`](/Users/naman/Documents/QBApps/qbapi/app/QBChannel.php#L104).

**Payment Processing**
- Order creation enters through [`app/Http/Controllers/OrderController.php`](/Users/naman/Documents/QBApps/qbapi/app/Http/Controllers/OrderController.php#L153) and the v2 controller path in [`app/Http/Controllers/v2/OrderController.php`](/Users/naman/Documents/QBApps/qbapi/app/Http/Controllers/v2/OrderController.php#L162), both of which call `Order::postOrder`.
- The payment rows themselves are built in [`app/Order.php`](/Users/naman/Documents/QBApps/qbapi/app/Order.php#L6385). This method already copies arbitrary `paymentType` strings, so `PAYMENT_BNPL` will flow if registration/account mapping is correct.
- Settlement endpoints are wired in [`app/Http/routes.php`](/Users/naman/Documents/QBApps/qbapi/app/Http/routes.php#L413) and handled by `OrderController@creditSettlement`, `@settleOrder`, `@postAdvancePayment`, `@postAdvanceDebit`.
- The main settlement logic is in [`app/Order.php`](/Users/naman/Documents/QBApps/qbapi/app/Order.php#L2581), [`app/Order.php`](/Users/naman/Documents/QBApps/qbapi/app/Order.php#L3489), [`app/Order.php`](/Users/naman/Documents/QBApps/qbapi/app/Order.php#L4410), and [`app/Order.php`](/Users/naman/Documents/QBApps/qbapi/app/Order.php#L17068).
- If BNPL is provider-backed rather than generic, there is already Snapmint-specific BNPL-like integration in [`app/snapMint.php`](/Users/naman/Documents/QBApps/qbapi/app/snapMint.php#L122) and gateway state in [`cloud_schema/globals.sql`](/Users/naman/Documents/QBApps/qbapi/cloud_schema/globals.sql#L6435).
