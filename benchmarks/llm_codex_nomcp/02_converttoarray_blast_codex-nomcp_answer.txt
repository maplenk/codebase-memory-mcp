**What Breaks If Return Type Changes To Object**

A change from array to object would break broadly and immediately.

- Any caller using `$data['x']` or `$rows[0]['x']` breaks with object access errors.
- Any caller passing the result into array-oriented code breaks: `new V($data)`, `array_column`, `array_merge`, helpers like `createArrayKey($array, ...)`, and many `count()/sizeof()` sites.
- Request-body entrypoints break first, because most controllers do `convertToArray(json_decode($result))` and then treat the payload as an associative array.
- DB-result consumers break next, because the codebase assumes query results become arrays of associative arrays.
- There is also a helper-internal risk: if you naively switch `json_decode(..., true)` to object mode, the sanitizer in [`app/helpers.php`](/Users/naman/Documents/QBApps/qbapi/app/helpers.php#L1298) no longer matches the data shape. Top-level objects would bypass sanitization, and arrays of objects would hit the scalar sanitization path incorrectly.
