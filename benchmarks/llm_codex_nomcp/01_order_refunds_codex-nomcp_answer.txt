**End-to-End Flow**
- Core refund logic lives in [app/Order.php:14561](/Users/naman/Documents/QBApps/qbapi/app/Order.php#L14561). It rejects missing `refundOrderID`, duplicate refund IDs, empty `refundsList`, and refunds whose summed payment amount would push gross bill below zero.
- It updates the original order in `orderSummary`/`orderDetail`, builds reverse accounting with [app/Account.php:1012](/Users/naman/Documents/QBApps/qbapi/app/Account.php#L1012) `reverseRounding` and [app/Account.php:1091](/Users/naman/Documents/QBApps/qbapi/app/Account.php#L1091) `orderReverse`, then inserts negative refund payments into `orderPayments` with `paymentRefundOrderID` linking them to the refund order.
- If chain-level invoices are enabled, it creates a separate refund invoice in `invoiceSummary`/`invoiceDetail`/`invoicePayments`; that refund invoice uses `orderID = refundOrderID` and `orderRefID = original orderID`.
- After DB writes, the controller dispatches side effects: stock or stock-ledger, accounting, loyalty, sales target, Unicommerce, Navision, IRD, EPOS, Zoho credit note, EasyRewardz, Nector, Mobiquest, and refund webhooks. Online orders additionally queue [app/Jobs/EcomVoidOrderPaymentRefundJob.php:13](/Users/naman/Documents/QBApps/qbapi/app/Jobs/EcomVoidOrderPaymentRefundJob.php#L13), which calls [app/OnlineOrder.php:11354](/Users/naman/Documents/QBApps/qbapi/app/OnlineOrder.php#L11354) `partiallyRefundEcomOrder`.

- Refund amount math is mostly request-driven. The server sums `refundPayments[].amount` for summary `voidAmount`, validates it against original gross bill, computes per-line `voidAmount` proportionally from original `productValue / quantityOrdered * voidQuantity`, and creates refund invoice lines as negative deltas between original and updated values. Loyalty is excluded from gross refund math; credit-note refunds create advance-payment/customer-account movements.
- Partial refunds are supported through `ORST_PARTIAL_CANCELLED`. Multi-tender is supported in the core order flow because `refundPayments[]` can contain multiple `paymentSubID`s, each written as its own negative payment row. No-charge and no-invoice-sale refunds are explicitly handled in invoice-number generation via refund prefixes and flags.
- The weak spots are in ecom refunds.
