There are two numbering paths in this repo, but the current server-generated path is handled by `App\Services\InvoiceNumberService`. It is called from the main order flow and refund flow when `generateInvoiceNumberFromServer = 1` and online billing is enabled: [InvoiceNumberService.php#L26](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/InvoiceNumberService.php#L26), [Order.php#L10891](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Order.php#L10891), [Order.php#L16536](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Order.php#L16536). The older device-side fiscal reset path still exists via `globalResetInvoice` and is managed by `FiscalYearManager::resetDeviceInvoiceNumbers`: [FiscalYearManager.php#L257](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/FiscalYearManager.php#L257).

Fiscal-year rollover works by setting each store’s `storeTransactionResetDate` to the new FY start date, applying the new prefixes, and then seeding sequence rows for that reset date. The generator chooses a sequence row by `(storeID, isNoCharge, isRefund, resetDate)`, using the store reset date once `storeTransactionResetDate <= today`; otherwise it uses the sentinel `1970-01-01`: [FiscalYearManager.php#L150](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/FiscalYearManager.php#L150), [FiscalYearManager.php#L454](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/FiscalYearManager.php#L454), [InvoiceNumberService.php#L205](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/InvoiceNumberService.php#L205), [InvoiceNumberService.php#L232](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/InvoiceNumberService.php#L232). On a new FY, the counter is reset by inserting or updating the new FY row with `lastInvoiceNumber = 0`, so the first generated invoice becomes `1`: [FiscalYearManager.php#L575](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/FiscalYearManager.php#L575), [FiscalYearManager.php#L629](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/FiscalYearManager.php#L629). The schema keeps separate counters for regular/no-charge and refund/non-refund, but not for `isNoInvoiceSale`; so RWI refunds get their own prefix while sharing the refund counter series. That is an inference from the service + unique key shape: [InvoiceNumberService.php#L72](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/InvoiceNumberService.php#L72), [InvoiceNumberService.php#L191](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/InvoiceNumberService.php#L191), [2025_11_06_000001_create_invoice_sequence_table.php#L45](\/Users\/naman\/Documents\/QBApps\/qbapi\/database\/migrations\/2025_11_06_000001_create_invoice_sequence_table.php#L45).

Compliance/business rules enforced in code are mostly around sequence integrity and prefix correctness: required prefix fields must exist, the fiscal-year date must be `YYYY-MM-DD`, store-level FY prefixes must be saved per store if the chain uses store prefixes, store prefixes must be unique within a chain/fiscal year, enabling server-generated numbering is blocked if stores are missing prefixes or historical duplicate invoice numbers exist, and manual invoice edits must start with the configured prefix: [InvoiceNumberService.php#L97](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/InvoiceNumberService.php#L97), [FiscalYearManager.php#L677](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/FiscalYearManager.php#L677), [FiscalYearManager.php#L694](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/FiscalYearManager.php#L694), [FiscalYearManager.php#L716](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/FiscalYearManager.php#L716), [Admin.php#L20891](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Admin.php#L20891), [Friday.php#L1418](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Friday.php#L1418).

ON DUPLICATE KEY UPDATE`, a non-`NULL` sentinel reset date, and retry-on-deadlock/lock-timeout logic: [InvoiceNumberService.php#L229](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/InvoiceNumberService.php#L229), [InvoiceNumberService.php#L273](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/InvoiceNumberService.php#L273), [2025_11_06_000001_create_invoice_sequence_table.php#L41](\/Users\/naman\/Documents\/QBApps\/qbapi\/database\/migrations\/2025_11_06_000001_create_invoice_sequence_table.php#L41), [DatabaseManager.php#L25](\/Users\/naman\/Documents\/QBApps\/qbapi\/app\/Services\/DatabaseManager.php#L25). Midnight rollover is split across two behaviors: server-generated numbering uses the order’s `posDate` passed into `generateInvoiceNumber`, so a late request with the old fiscal date stays on the old bucket, while the device-side reset activates based on `CURRENT_DATE` and forces device sync around midnight.
